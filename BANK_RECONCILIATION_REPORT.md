# 月結單對帳與銀行帳戶功能完整度報告

經過對 GitHub 儲存庫 `kitahuman/mingtat-erp` 程式碼的全面檢視，以下是關於「銀行帳戶設定」與「月結單對帳（Bank Reconciliation）」功能的開發進度、現況分析與建議報告。

## 系統架構與資料庫模型設計

系統在資料庫層面已經為銀行帳戶與交易記錄建立了完整的基礎架構。在 `schema.prisma` 中定義了 `BankAccount` 模型，用於儲存銀行帳戶的基本資訊，如帳戶名稱、銀行名稱、帳號與幣種等。同時，`BankTransaction` 模型負責記錄從月結單匯入的每一筆交易明細，包含日期、描述、金額、借貸方向、餘額與參考號碼，並透過 `match_status` 欄位追蹤該筆交易的配對狀態（未配對、已配對或已排除）。這兩個模型之間透過 `bank_account_id` 建立了明確的一對多關聯。

然而，在現有架構中存在一個潛在的資料一致性問題。雖然 `PaymentIn`（收款記錄）和 `PaymentOut`（付款記錄）模型中都包含了 `bank_account` 欄位，但這些欄位被定義為自由文字輸入（`String? @db.VarChar(100)`），而非關聯到 `BankAccount` 模型的外部鍵。這種設計斷層會導致系統無法嚴格確保收款與付款記錄準確對應到系統內已建立的特定銀行帳戶。

## 後端服務與 API 實作進度

在後端實作方面，銀行帳戶管理（`bank-accounts` 模組）已經提供了完整的 CRUD（新增、讀取、更新、刪除）操作端點，並在刪除帳戶前加入了防呆機制，確保不會誤刪已有交易記錄的帳戶。

月結單對帳（`bank-reconciliation` 模組）的後端邏輯也已基本成型。系統支援透過 CSV 匯入交易記錄，並在匯入過程中執行重複檢查，避免匯入相同帳戶、日期、金額與描述的重複資料。匯入完成後，系統會自動觸發配對邏輯（`autoMatch`）。自動配對的規則包含比對參考號碼與金額，以及在沒有參考號碼時，尋找日期在前後三天內且金額完全吻合的唯一記錄。此外，系統也提供了尋找候選配對的 API 端點，允許使用者在手動配對時，尋找前後十五天內金額誤差在百分之十以內的相關收付款記錄。

儘管後端功能已經串通，但由於前述的資料庫設計問題，自動配對邏輯在尋找相符的 `PaymentIn` 或 `PaymentOut` 時，並未限制必須屬於同一個銀行帳戶。這意味著系統有可能將 A 銀行的交易錯誤配對到 B 銀行的付款記錄上，這是一個需要優先解決的邏輯漏洞。

## 前端介面與使用者體驗

前端介面已經涵蓋了銀行帳戶管理與月結單對帳的主要流程。在銀行帳戶設定頁面（`/settings/bank-accounts`），系統使用了行內編輯資料表（`InlineEditDataTable`），讓使用者可以直觀地新增與維護銀行帳戶資訊。

月結單對帳主頁面（`/bank-reconciliation`）提供了切換銀行帳戶的下拉選單，並在頂部顯示總交易筆數、已配對、未配對筆數及金額的摘要資訊。資料表列出了所有交易記錄，並提供了「配對」、「取消配對」與「排除」的操作按鈕。然而，目前的資料表缺乏狀態與日期篩選功能，使用者在面對大量交易時，難以快速過濾出需要處理的未配對記錄。

在匯入月結單的介面（`ImportModal.tsx`）中，雖然支援上傳 CSV 檔案並提供預覽功能，但其解析邏輯過於僵化。系統目前硬編碼了日期的解析格式（DD/MM/YYYY）以及欄位的固定順序（日期、描述、金額、餘額、參考號）。考慮到不同銀行（如匯豐、中銀、恒生等）提供的 CSV 格式往往存在顯著差異，這種設計在實際應用中極易導致匯入失敗或資料解析錯誤。

手動配對介面（`MatchModal.tsx`）在列出候選付款記錄時，目前主要顯示關聯的工程名稱（`project_name`）。然而，許多付款記錄（`PaymentOut`）實際上是關聯到公司（供應商）或糧單，而非特定工程。這導致使用者在手動配對時，無法獲得足夠的資訊（如收款人名稱或付款描述）來做出正確的判斷。

## 綜合評估與改善建議

整體而言，月結單對帳功能的核心流程已經建立並可以初步運作，功能完整度約為 75%。為了達到生產環境的標準，建議優先處理以下改善項目：

| 優先級 | 改善項目 | 問題描述 | 建議解決方案 |
|---|---|---|---|
| 高 | 修復收付款與銀行帳戶的關聯斷層 | `PaymentIn` 和 `PaymentOut` 的 `bank_account` 欄位為自由文字，導致自動配對時可能發生跨帳戶的錯誤配對。 | 將 `schema.prisma` 中的 `bank_account` 改為 `bank_account_id` 外部鍵，並更新前端表單使用下拉選單選擇銀行帳戶。 |
| 高 | 強化 CSV 匯入的容錯與格式支援 | 匯入解析邏輯過於僵化，無法適應不同銀行多變的 CSV 格式，容易導致匯入失敗。 | 實作靈活的 CSV 欄位對應介面，讓使用者在匯入時指定欄位；或在後端針對主要銀行實作專屬的解析器。 |
| 中 | 完善對帳頁面的篩選功能 | 月結單對帳主頁面缺乏狀態與日期篩選器，使用者難以快速找出未配對的交易。 | 在前端加入「狀態」與「日期範圍」的篩選下拉選單，提升操作效率。 |
| 中 | 改善手動配對介面的顯示資訊 | 手動配對時，候選付款記錄未顯示供應商名稱或詳細描述，導致使用者難以辨識。 | 修改 `MatchModal.tsx`，優先顯示供應商名稱（`company.name`）或付款描述（`payment_out_description`）。 |
